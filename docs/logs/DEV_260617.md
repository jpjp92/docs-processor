# DEV_260617

## Summary

PDF 텍스트 치환 기능을 메인 앱에 베타 형태로 연결했고, 실제 PDF 테스트를 통해 현재 방식의 한계와 제품화 방향을 정리했다.

현재 결론:

- 기존 PDF content stream을 직접 치환하는 방식은 실험 기능으로 유지한다.
- 일반 사용자가 쓰는 기본 편집 UX는 `pdf-lib` 기반 overlay replacement 방식이 더 적합하다.
- 드래그 선택 후 페이지 위에 편집 버튼을 띄우는 UX로 전환하는 방향이 좋다.

## Current Implementation

추가된 서버 API:

- `POST /api/pdf-edit/inspect`
  - 현재 페이지의 PDF 내부 텍스트 후보를 추출한다.
  - 후보별 텍스트, 폰트, 글리프 수, 레이어 수, 대략적인 bounds를 반환한다.
  - 후보별 `supportedChars`를 반환해 현재 embedded subset font에서 실제로 쓸 수 있는 글자를 UI가 알 수 있게 했다.
- `POST /api/pdf-edit/apply`
  - 선택한 후보를 지정한 새 텍스트로 치환한다.
  - 성공 시 수정된 PDF binary를 반환한다.

핵심 라이브러리:

- `lib/pdf-edit/selection-replace.ts`

메인 UI:

- 오른쪽 패널 상단에 `PDF 텍스트 치환 실험` 섹션 추가
- 후보 찾기
- 후보 선택
- 새 텍스트 입력
- 글리프 수 및 폰트 지원 문자 검사
- 수정 PDF 다운로드
- 자동 다운로드가 막히는 경우를 대비한 `생성된 PDF 다시 받기` 링크 제공

## Route Change

초기에는 치환 API를 다음 경로로 구성했다.

```text
/api/pdf-edit/replace
```

하지만 Next dev 서버에서 해당 경로가 Server Action 요청으로 오인되어 404가 발생했다.

```text
Server action not found.
```

그래서 적용 API 경로를 다음으로 변경했다.

```text
/api/pdf-edit/apply
```

변경 후 실제 PDF 기준 `inspect`, `apply` 모두 200 응답을 확인했다.

## Direct Replacement Limits

현재 직접 치환 방식은 다음 조건을 만족해야 한다.

- PDF가 실제 텍스트 객체를 포함해야 한다.
- text run을 추출할 수 있어야 한다.
- 해당 폰트에 `ToUnicode` CMap이 있어야 한다.
- 기존 글리프 수와 새 글리프 수가 같아야 한다.
- 새 글자가 해당 PDF embedded subset font에 이미 포함되어 있어야 한다.

예시:

- 후보 `통권`은 2글리프
- 입력 `헬로`도 2글리프
- 하지만 `/F1` 폰트에 `헬`, `로` 글리프가 없어 실패

오류:

```text
Missing glyph(s) in /F1: 헬, 로
```

이 문제는 글리프 수 검사만으로는 사용자에게 충분히 설명되지 않는다.

조치:

- 후보별 `supportedChars`를 API 응답에 포함
- UI에서 현재 폰트에 없는 글자를 사전 표시
- 미지원 글자가 있으면 `수정 PDF 다운로드` 버튼 비활성화

## Layout Accuracy Limits

정밀 치환도 시각적으로 항상 안전하지 않다.

이유:

- PDF 텍스트는 일반 문자열이 아니라 `TJ` 배열, 글리프 코드, 자간 보정값으로 구성된다.
- 현재 구현은 글리프 코드만 같은 개수로 바꾼다.
- 새 글자의 실제 폭이 기존 글자 폭과 다르면 같은 글리프 수여도 겹치거나 어긋날 수 있다.
- 원본 자간 보정값은 그대로 유지되므로 새 글자 조합에 맞지 않을 수 있다.
- 그림자, 투명도, 반복 레이어가 있는 경우 일부 레이어만 바뀌거나 잔상이 남을 수 있다.

따라서 현재 기능은 다음처럼 정의하는 것이 정확하다.

```text
PDF content stream의 기존 text run을 조건부로 치환하는 실험 기능
```

일반 사용자용 자유 편집 기능으로 보기에는 제약이 크다.

## Product Direction

사용자가 기대하는 PDF 텍스트 편집 UX는 다음 방식이 더 적합하다.

```text
overlay replacement
```

예상 흐름:

1. 사용자가 PDF 페이지에서 수정할 영역을 드래그한다.
2. 영역 근처에 `편집` 플로팅 버튼을 띄운다.
3. 편집 팝오버에서 새 텍스트를 입력한다.
4. 기존 텍스트 영역을 흰색 또는 추정 배경색 사각형으로 가린다.
5. `pdf-lib` + `fontkit`으로 한글 폰트를 embed한다.
6. 같은 위치에 새 텍스트를 그린다.
7. 수정 PDF를 다운로드한다.

장점:

- 사용자가 자유롭게 한글을 입력할 수 있다.
- embedded subset font에 없는 글자도 처리할 수 있다.
- 글리프 수 제한이 없다.
- 제품 UX가 훨씬 자연스럽다.

단점:

- 원본 텍스트 객체를 실제로 삭제하는 것은 아니다.
- 배경이 복잡하면 가림 사각형이 티 날 수 있다.
- 원본 폰트와 동일 폰트를 모르면 완전한 디자인 일치는 어렵다.
- PDF 검색/복사 시 원본 텍스트가 남을 수 있다.

## Recommended Feature Split

### 기본 편집 모드

- `pdf-lib` overlay 방식
- 자유 입력 지원
- 한글 폰트 embed
- 드래그 선택 기반 플로팅 편집 UI
- 사용자가 보기에는 가장 편한 방식

### 정밀 치환 모드

- 현재 content stream 직접 치환 방식
- 고급/실험 기능으로 유지
- 기존 폰트 글리프만 사용 가능
- 같은 글리프 수여도 시각적 폭 차이로 틀어질 수 있다는 경고 필요

## Rollback (2026-06-18)

직접 치환 방식은 다음 한계로 일반 편집 기능으로 부적합하다고 결론냈다.

- 기존 embedded subset 폰트에 없는 글자는 치환 불가 (`Missing glyph`)
- 같은 글리프 수여도 글자 폭 차이로 시각적 어긋남 발생
- TJ 자간 보정값/반복 레이어로 인해 깔끔한 치환 보장 불가

따라서 치환 **코드와 UI는 전부 제거(롤백)** 하고, 분석/계획 문서만 기록으로 남긴다.

제거한 항목:

- `lib/pdf-edit/selection-replace.ts`
- `app/api/pdf-edit/inspect/route.ts`
- `app/api/pdf-edit/apply/route.ts`
- `components/pdf-workspace.tsx`의 `PDF 텍스트 치환 실험` 패널 (HEAD로 복원)
- `app/globals.css`의 해당 패널 스타일 (HEAD로 복원)

남긴 항목:

- 본 로그 및 `docs/PDF_TEXT_EDIT_INTEGRATION_PLAN.md`, `docs/PDF_TEXT_OBJECT_EDITING.md`
- 로컬 랩 스크립트(`scripts/pdf-*`)와 `tmp/pdf-text-object-lab/` 산출물 (gitignore, 참고용)

향후 방향: 직접 치환 대신 `pdf-lib` overlay 방식을 기본 편집 UX로 검토한다.
